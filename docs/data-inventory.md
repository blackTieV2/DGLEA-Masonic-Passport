# DGLEA Masonic Passport — Data Inventory

This document inventories the data classes handled by the DGLEA Masonic Passport backend and Android app. It is a living document and should be updated when the schema or API changes.

## Data Model Source

- Backend ORM schema: `backend/prisma/schema.prisma`
- PDF/export service: `backend/src/modules/exports/exports.service.ts`

## Data Classification

| Classification | Definition | Examples |
|----------------|------------|----------|
| **Public / reference data** | Non-personal information that can be shared broadly. | Static reference pages, passport templates, milestone descriptions. |
| **Internal Lodge data** | Lodge operational data; not secret but scope-limited. | Lodge name/number, district, meeting location, secretary contact. |
| **Personal data** | Information relating to an identified or identifiable living individual. | Name, email, phone, preferred name, Firebase UID, lodge affiliation. |
| **Sensitive progress / mentoring records** | Detailed records about a Brother’s progression and mentor interactions. | Mentor notes, approval notes, session summaries, reflections, visitation notes, sign-off notes. |
| **Operational / security data** | Data needed to run and secure the system. | Audit events, role assignments, notifications, authentication metadata. |
| **Secrets / credentials** | Must never be committed or exposed. | Database URL, Firebase service account keys, Android signing keystore/passwords. |

## Data Flows

1. **Sign-in:** Firebase Auth validates email/password and returns an ID token. The Android app sends the ID token to the backend in the `Authorization: Bearer` header. The backend verifies the token and looks up the `User` record by `firebaseUid`.
2. **Profile load:** Backend returns Brother profile, lodge, and degree progress. Data is held in memory by the app and not persisted locally beyond the session cache.
3. **Mentor/approver actions:** Verifiers submit reviews, notes, and approvals; these are stored in the backend DB and recorded in `AuditEvent`.
4. **PDF export:** Backend generates a PDF or HTML document on demand, containing personal and progress data. The file is returned to the app and may be saved or shared by the user via the Android system share sheet.
5. **Reference content:** Static reference pages are read from the DB and displayed in the app. They do not contain personal data.

## Notable Gaps

- The app does not currently encrypt local session cache beyond Android OS protections.
- No automatic purge or retention policy is configured for audit events, notifications, or exported files.
- PDF exports are not watermarked or expiry-dated.
